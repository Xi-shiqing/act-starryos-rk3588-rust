# 在 StarryOS + Orange Pi 5 Plus（RK3588） 上跑通 ACT 推理的上板调试历程

本文记录把 ACT 推理从「服务器侧模拟」推进到「StarryOS + 真 RKNPU 上板运行」的全过程，重点是**遇到的障碍、定位方法与最终解法**，供复现与后续移植参考。目标硬件为 Orange Pi 5 Plus（RK3588）。

## 起点

在上板之前已具备：
- 用 tgoskits 编出的、含 rknpu 驱动的 StarryOS Orange Pi 5 Plus 内核（`Image`）与设备树（`dtb`）；
- 注入了 aarch64 glibc 运行时、并塞入推理载荷 `/act_rknn`（`act-rknn` + fp16 模型 + 56 帧）的 rootfs；
- 用 qemu-user 验通的用户态链路（`act-rknn` 能加载 librknnrt 跑到 `rknn_init`，仅因 x86 无 NPU 而停）；
- 该模型在 Ubuntu 22.04 + 同一块 RK3588 NPU 上实测过，结果为：判向 46/56，右转召回 18/19，约 25 ms/帧。

目标：做出一张烧卡即跑的 TF 卡镜像，让开发板从 StarryOS 启动并完成同样的推理。

## 路线选择

两条上板路径：
1. ostool 串口引导（tgoskits 官方测试用）：在 PC 上用 `cargo starry uboot` 经 U-Boot 把内核+dtb 载入内存 `booti`；
2. 自包含整盘镜像：把引导块、内核、dtb、rootfs 拼成一张可 `dd` 的整盘卡，插卡即从板载 U-Boot 启动。

为了让烧录、运行尽量简单，选择路线 2。这一选择也带来了后文若干与板载 rockchip U-Boot 相关的坑可见后文。

## 坑一：整盘可启动镜像如何拼

RK3588 的启动链为：BootROM → idbloader（一级，固定在卡的扇区 64）→ u-boot.itb（二级，扇区 16384）→ U-Boot 扫描分区上的 `extlinux.conf` → 加载内核启动。

镜像缺的只是 RK3588 的引导块。做法：从开发板自身那张能启动的卡里，把扇区 64..32768 的 idbloader + u-boot.itb 原样 `dd` 出来，作为构建输入。随后用 `sgdisk` 组装 GPT 整盘：

```
扇区 64      idbloader.img          （从板上抠出）
扇区 16384   u-boot.itb             （从板上抠出）
分区 1(ext4) StarryOS 根文件系统    （从扇区 32768 起）
```

校验引导块魔数无误（idbloader 头 `RKNS`、u-boot.itb 头 `d00dfeed`）即可。

## 坑二：StarryOS 挂错了根（ramdisk vs 分区）

最初的交付设计把 rootfs 当作 ramdisk/initrd 投递。但实际上板日志显示 StarryOS 并不使用 initrd：

```
ax_fs_ng::root: only one supported filesystem partition is available; using it as root
selected root device: disk0 partition 1 (..., fs=ext4)
```

查 StarryOS 源码确认：`axfs-ng` 在启动时扫描磁盘分区、挑选唯一的 ext4 分区作为根挂载；其 PID1 是内核内置的 `/bin/sh -c <编进内核的 init.sh>`（`starryos/src/main.rs`），并不会去读 initrd，也不走 rootfs 里的 `/sbin/init`。

因此整盘镜像里的「那个 ext4 分区」必须本身就是完整的根文件系统（带 glibc 运行时、`/bin/sh`、以及 `/act_rknn` 载荷），而不是一个只放了内核和 rootfs 镜像文件的「boot 分区」。改成把真 rootfs 直接铺成分区 1 后，根挂载正确。

## 坑三：挂载后 `Syncing filesystem` 死锁

根挂载成功后，启动卡死在：

```
rsext4::ext4::sync: Syncing filesystem...
```

分析：`axfs-ng` 在挂载根之后会调用一次 sync，把超级块等元数据写回 SD；而在这一步挂死。同一时段，开发板自身那张 Ubuntu 卡也出现了 `Buffer I/O error` / 写超级块失败 —— 指向该平台 SD 写入路径本身的脆弱（StarryOS 的 dwmmc 写无超时，于是死等而非报错）。

关键观察：推理程序 `act-rknn` 只读模型/图片、输出到串口，不写盘。于是只要让根文件系统只读挂载，就能彻底绕开这次以及后续任何对 SD 的写入。查 StarryOS 源码 `fs/ext4/rsext4/fs.rs`：

```rust
match device_has_error_state(&dev) {
    Ok(true)            => mount_readonly_no_replay(dev),   // 只读、不重放日志
    Ok(false)           => rsext4::mount(dev),              // 读写 → 会做 sync
    Err(EUCLEAN)        => mount_readonly_no_replay(dev),
}
```
而 `mount.rs` 里那次会死锁的 sync 受 `if !options.readonly` 守卫 —— 只读挂载会跳过它。

解法：在构建镜像的最后一步，用 `debugfs -w -R 'ssv state 2'` 给根文件系统超级块置上 `EXT4_ERROR_FS` 标志，逼 StarryOS 走只读挂载分支。上板验证：不再卡 sync，`act-rknn` 真正开始执行（运行了 21 秒）。

> 附带处理：该 rootfs 由较新版 e2fsprogs 创建、带 `orphan_file` 特性，而 rsext4 不维护它；
> 将 ext4 特性集对齐到该平台 known-good 的集合（去 `orphan_file`、保留 `metadata_csum` 等）后更稳。

## 坑四：看不到用户态输出，且程序 Rust panic

只读挂载打通后，`act-rknn` 执行了 21 秒，但退出码为 25856（= 101 << 8，即 Rust panic），且串口上看不到程序的任何输出。日志里两行是线索：

```
Failed to bind console tty: NoSuchDevice
/dev/console has no serial TTY binding
```

即：用户态 `/dev/console` 没有绑定到任何串口设备，于是程序 `println!` 写向一个不可用的 stdout，Rust 标准库在写失败时直接 panic。所以 看不到输出 和 panic 很可能是同一个根因。

## 坑五：console 被板载 U-Boot 强制成 `tty1`

StarryOS 选择控制台串口的逻辑在 `platforms/somehal/src/boot_console.rs`：从内核 cmdline 的 `console=` 里取最后一个硬件串口（如 `ttyS2`），`console=tty1`（虚拟终端，非硬件串口）会被忽略；若一个硬件串口都没匹配到，则 `/dev/console` 不绑定。

而我们上板时内核拿到的 cmdline 是：

```
console=tty1 console=tty1 root=UUID=3ab40e80-... cma=128M cgroup_enable=...
```

没有 `console=ttyS2`。但我们交付的 dtb 里 `/chosen/bootargs` 明明写了 `console=ttyS2,1500000`，extlinux 的 `append` 也写了。逐一排查后定位到板载 rockchip U-Boot 的行为：

- 它在 `booti` 阶段会对 cmdline 做一次「android 改写」：删掉已有的 `console=`、强行塞入它自己的控制台 `console=tty1`，并补上 `cma` / `cgroup` / `ubootpart` / `androidboot.fwver` 等。
- 触发这条改写路径的，是 dtb 里自带的 `/chosen/bootargs`。

对照证据：开发板原版 Ubuntu 的启动 cmdline 是它 extlinux `append` 的原样（`console=ttyS2,1500000` 得以保留，也没有那一堆 android 参数）—— 因为 Ubuntu 用的 dtb 没有烤死的 `/chosen/bootargs`。

期间两次失败的尝试，也印证了这一点：
- 在 `init.sh` 里把输出重定向到 `/dev/ttyS2`：无效。该 tty 的后端只有在 `/dev/console` 绑定流程中才会被启动，直接写它到不了串口。
- 改用 `boot.scr` 显式 `setenv bootargs '...console=ttyS2...'` 再 `booti`：U-Boot 仍在 `booti` 时把 console 改回了 `tty1`。说明从 SD 卡侧（无论 extlinux 还是 boot.scr）都压不过这条改写。

最终方法：把我们 dtb 里的 `/chosen/bootargs` 属性删掉（用一段 Python 按 FDT 标准把该属性的 token 全部改写为 `FDT_NOP`，不破坏结构、文件大小不变），再像 Ubuntu 那样用 extlinux 的 `append` 提供 `console=ttyS2,1500000`。这样 U-Boot 不再走 android 改写、原样采用 append，`console=ttyS2` 得以保留：

```
/dev/console bound to ttyS2
Welcome to Starry OS!
==== ACT NPU inference start ====
rknn_init OK
...
summary: 56/56 frames, avg inference 26.6 ms
==== ACT NPU inference end (exit=0) ====
root@starry:/root #
```

控制台绑定成功，输出可见，panic 一并消失（exit=0）。

## 结果

| 指标 | StarryOS + 真 NPU | 对照：普通 Linux + 同一 NPU |
|---|---|---|
| 完成帧数 | 56/56 | — |
| 判向 vs 真值 | 46/56（82.1%） | 46/56 |
| 右转召回（19 帧） | 18/19 | 18/19 |
| 判向 vs fp32 模拟器 | 55/56 | — |
| 平均时延 | 26.6 ms/帧 | 24.7 ms/帧 |

StarryOS 上的结果与普通 Linux 真机实测**逐项一致**，证明 StarryOS 真 NPU 推理可复现、fp16 量化近乎无损。少数与真值不一致的帧均为左右轮速差极小（约 ±0.001~0.002）的临界样本，与 fp16 数值噪声同量级。

## 复现时要注意的

1. 自包含整盘镜像：从板上抠出 idbloader + u-boot.itb，拼 GPT（引导块@扇区 64 + ext4 根分区@扇区 32768）。
2. 真 rootfs 当根分区（非 ramdisk）：StarryOS 扫盘挂唯一 ext4 分区为根，PID1 为内核内置 `/bin/sh -c init.sh`。
3. 置 `EXT4_ERROR_FS` 标志逼只读挂载：跳过挂载后对 SD 的 sync 写（该平台写入路径会死锁），推理只读不写盘。
4. dtb 删 `/chosen/bootargs`：避免板载 rockchip U-Boot 改写 console，使 extlinux append 的 `console=ttyS2` 存活，从而绑定串口、看到用户态输出。

构建脚本：`app/build_sd_image.sh`；引导块：`bootloader/bootblob.bin.gz`；去 bootargs 的 dtb：
`app/orangepi-5-plus.nobootargs.dtb`；结果存档：`RESULT_starryos_board.md`。
