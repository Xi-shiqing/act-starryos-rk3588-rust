# StarryOS + Orange Pi 5 Plus（RK3588） ACT 推理烧录与运行手册

本镜像是一张自包含、可直接烧录的 TF 卡镜像：烧入后插卡上电，开发板即从 StarryOS 启动，在 RKNPU 上加载 ACT（fp16）模型，对 56 个明确转向帧做推理，并通过调试串口打印每帧的判向结果（向左 / 向右）与耗时。无需联网亦不依赖任何外部预制系统。


## 1. 准备

硬件：
- Orange Pi 5 Plus 开发板 + USB-C 电源；
- TF 卡一张，容量 ≥ 8 GB（虽然 Orange Pi 5 Plus 让用 16 GB 以上的卡，但是我们实测 8 GB 的卡能跑这个镜像）；
- 读卡器；
- USB 转 TTL 串口模块（CH340 / CP2102 / FT232 均可），3 根杜邦线（母对母）；
- 一台电脑。

软件包内容：
| 文件 | 说明 |
|---|---|
| `starryos-act-orangepi5plus-sd.img.gz` | 待烧录的整盘镜像（gzip 压缩，约 99 MB） |
| `烧录与运行手册.md` | 本文档 |
| `EXPECTED.csv` | 56 帧的标准判向（结果核对用） |
| `SHA256SUMS` | 解压后镜像的校验和 |



## 2. 烧录镜像

方式一、balenaEtcher
1. 安装 balenaEtcher（https://etcher.balena.io/）。
2. 读卡器插入电脑，插入 TF 卡。
3. Etcher 中：`Flash from file` 选择 `starryos-act-orangepi5plus-sd.img.gz`（`.gz` 无需解压）→ `Select target` 选择 TF 卡（**注意核对容量，勿选到本机硬盘**）→ `Flash`。
4. 出现 `Flash Complete!` 即完成。若 Windows 弹出「需要格式化磁盘」，请**取消**（卡已是 Linux 格式，Windows 不识别属正常）。


方式二、命令行（Linux / macOS）
```bash
gunzip starryos-act-orangepi5plus-sd.img.gz
# 将 /dev/sdX 替换为实际 TF 卡设备，务必核对，勿写错盘
sudo dd if=starryos-act-orangepi5plus-sd.img of=/dev/sdX bs=4M conv=fsync status=progress
sync
```

可选校验：`sha256sum -c SHA256SUMS`（需先解压出 `.img`）。



## 3. 连接调试串口

Orange Pi 5 Plus 的调试串口为板边的 3 针 UART（丝印通常为 `GND / TXD / RXD` 或 `UART`、`DEBUG`）。按「TX 接 RX 交叉」连接，不接电源线（详见 Orange Pi 5 Plus 官方手册）



## 4. 打开串口终端

串口参数：波特率 1500000，8N1，无流控。

- Windows（PuTTY）：设备管理器 →「端口 (COM 和 LPT)」查到 USB-TTL 的 COM 号；PuTTY 中 Connection type 选 `Serial`，Serial line 填 `COMx`，Speed 填 `1500000`，Open。
- Linux：`sudo screen /dev/ttyUSB0 1500000`（退出：`Ctrl+A` `K` `y`）。
- macOS：`ls /dev/tty.usb*` 找到设备名后 `sudo screen /dev/tty.usbserial-XXXX 1500000`。

如需保存完整日志：PuTTY 可在 `Session → Logging → All session output` 设定日志文件；
`screen` 可用 `screen -L`。



## 5. 上电与运行

1. 将烧好的 TF 卡插入开发板 SD 卡槽。
2. 给板子上电。
3. 串口终端会输出 U-Boot → 内核 → StarryOS 的启动信息。

启动过程中以下两条为预期信息，并非错误：
- `ext4 filesystem is in error state; mounting read-only without journal replay`
  —— 镜像有意将根文件系统标记为只读挂载（推理只读不写盘），以规避该平台 SD 写入路径上的一个挂死问题。
- `/dev/console bound to ttyS2` —— 用户态控制台已正确绑定到调试串口。

启动完成后显示 `Welcome to Starry OS!`，随后自动运行推理，逐帧打印：
```sh
==== ACT NPU inference start ====
rknn_init OK
frame_000007.jpg: left_vel=.. right_vel=.. diff=.. decision=left (xx ms)
...
summary: 56/56 frames, avg inference xx.x ms
==== ACT NPU inference end (exit=0) ====
```
若系统停在 `root@starry:/root #` 提示符而未自动运行，可手动执行：
```sh
sh /act_rknn/init.sh
```



## 6. 结果核对

每帧输出末尾的 `decision=left` / `decision=right` 即模型判定的转向方向。
随包的 `EXPECTED.csv` 中 `gt_dir` 列为各帧的人工标注方向（真值）。

参考指标（与该模型在普通 Linux + 同一 NPU 上的实测一致）：
- 判向 vs 真值：约 46/56（82%），其中右转召回 18/19；
- 判向 vs fp32 模拟器：55/56（NPU fp16 量化近乎无损）；
- 单帧耗时约 26 ms。

少数与真值不一致的帧，集中在左右轮速差极小（约 ±0.001~0.002）的临界样本上，
与 fp16 在真硬件上的数值噪声同量级，属已知现象。



## 7. 故障排查（仅供参考）

| 现象 | 处理 |
|---|---|
| 串口无任何输出 | 检查三根线是否接牢；将 TX/RX 对调试一次；确认波特率为 `1500000`；确认所选 COM/设备正确；确认板子已上电 |
| 输出为乱码（方块/问号） | 波特率不对，确认设为 `1500000` |
| 烧录时 Windows 提示格式化 | 选择「取消」，烧录已完成即可拔卡 |
| 板子未从 TF 卡启动 | 断电后确认卡插好再上电；必要时清除板上其它存储介质的首扇区以避免抢占启动 |
| 卡在某一行不再继续 | 保存从开头到该行的完整串口日志以便定位 |



## 8. 镜像构成（简述）

| 区域 | 内容 |
|---|---|
| 扇区 64 / 16384 | RK3588 idbloader / u-boot.itb（一二级引导） |
| 分区 1（ext4，标签 `rootfs`） | StarryOS 根文件系统：`/boot/Image`（含 rknpu 驱动的内核）、`/boot/dtb/`、`/boot/extlinux/`、glibc 运行时、`/act_rknn`（推理程序 + fp16 模型 + 56 帧 + 启动脚本） |

构建脚本见 `../app/build_sd_image.sh`；关键设计与排障过程见 `../上板调试历程.md`。
